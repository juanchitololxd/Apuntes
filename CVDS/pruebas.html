Pruebas

Ventajas

  • Ayuda a encontrar defectos de software
  • Entre mas pruebas mas confiabilidad en el software
  • Al modificar el software, ayuda a encontrar errores que son introducidos en nuevo codigo

Desventajas

Piramide del test de automatización

  1. UI Tests: Todo el sistema
  2. Integration tests
  3. Unit tests

Unit tests

Es una prueba de una funcionalidad del código de manera aislada

ventajas:

  • Asegura calidad del código
  • Reduce los problemas y tiempos dedicados a la integración
  • Documentación del codigo

Retos

  • No garantiza la ausencia de errores
  • No detecta errores del sistema probado en conjunto
  • Estan sujetos abuenas practicas de codificación

Principio FIRST

Las pruebas deben ser

  • Fast
  • Isolated
  • Repeteable
  • Self-checking: Automático
  • Timely: Oportunas

Formas de abordar las pruebas

  • Test Driven Development - TDD
  • El otro enfoque

Enfonque para abordar las pruebas

  • Pensar como abordaría cada clase
  • Se programa la unidad
  • Se programa las pruebas con patron AAA
  • Se ejectuan las pruebas

Test Driven Development - TDD

  • Escribir la prueba
  • Validar que la prueba falla
  • Escribe el codigo minimo para que la pruebas pase
  • Refactoriza lo necesario
  • Valida que todo pase
  • Vuelve a escribir la prueba y se repite la vaina

Patron AAA

  • Arrange: Preparar valores iniciales
  • Act: Ejecutar lo probado
  • Assert: Validar el reultado

Coverage

Es el codigo minimo que exigen para que se consideren las pruebas como validas

Hay apps donde uno puede programar que las pruebas prueben alemnos un n% del codigo

Coverage = N° Lineas de codigo testeadas * 100 / N° Lineas de codigo totales

Errores novatos

  • El nombre del test no es lo suficientemente descriptivo
  • No se sabe que hará la unidad
  • Usar los mismos test unitarios para las pruebas de integración
  • No se elimina codigo muerto (variables creadas innecesariamente)

Antipatrones de las pruebas unitarias

  • El mentiroso: Muchos asserts pero ninguno prueba lo que debe probar
  • Setup Excesivo: Mucha configuración de la data hace que no se vea lo que se quiere probar
  • Test Dios: Prueba todo
  • El que siempre funciona: assertTrue(true == true)
  • El libre albedrío: Si una función llama otras funciones en base a condiciones, se debe crear otra función que pruebe ese flujo
  • Test con comentarios: Un test por su forma de ser escrito no debería de necesitar comentarios

Tipos de pruebas

  • Caja blanca: Diseño de pruebas en base a las funcionalidades del codigo fuente
  • Caja negra: Diseño de pruebas en base a los requerimientos funcionales
  • Funcionales: Verifican que el prd cumple con los especificado en las reglas de negocio
  • No funcionales: Validan aspectos de calidad (Rendimiento, usabilidad y fiabilidad)

Clases de equivalencia

Conjunto de datos para los cuales el comportamiento del sistema es el mismo.

Dominio

Es el universo de los datos con los que se puede probar

Analisis de frontera
