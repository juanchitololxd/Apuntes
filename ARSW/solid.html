SOLID


Conjunto de buenas practicas diseñados por Robert J. Martin. Creador de los libros *Clean-code* y *Clean Architecture* con el fin de generar código legible y comprobable


Single Responsaiblity


Indica que una clase solo debe realizar una acción. Si se requiere que una clase tenga otra funcionalidad en otro sector, es mejor crear otra clase. Un ejemplo es crear una clase aparte para la persistencia de un objeto en lugar de acoplar ese código en una misma clase.




Open Close Principle


*Las clases deben estar abiertas a la extensión (adición de funcionalidades) y cerradas a la modificación (Cambiar el código de una clase existente)*


Esto usualmente se logra por medio de interfaces y clases abstractas




Liskov Sustitution


Todas las subclases deben ser sustituibles por una clase base (es herencia básicamente).


Interface Segregation


*Muchas interfaces específicas del cliente son mejores que una interfaz de propósito general*




Dependency inversion


Los módulos de alto nivel no deben depender de los de bajo nivel
